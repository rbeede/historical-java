2.0  Description

 

Stacks, Queues, and Lists are good, unique ADT’s for storing and manipulating different forms of data. So, to explore the various uses, as well as properties, for each, we will implement a simple system to handle patients in an emergency room, known as the Emergency Room Patient Manager, or ERPM. The system we develop could also be easily adapted to many other situations.

First, we will maintain a queue of patients waiting for treatment. We will also maintain a queue of doctors available to see patients. Patients and doctors will both be identified by an id number, therefore our queues must only store their id numbers. Both of these queues will be read from files, called PatientQ.txt and DoctorQ.txt, respectively. Each entry will be on a separate line of the file, and the first line of each file will be the head of each queue (ie. The first patient to be seen, or the first doctor available.)

We will also maintain a database for patient information. We will use a List ADT for this, and each node of the List will store the data for one patient. The data stored is the patient’s ID number, as well as a Stack which will store their patient history.  Each history element of the stack will have a numeric code representing a diagnosis, a Date for the diagnosis, and the Doctor ID of the doctor who made the diagnosis. True to the nature of a Stack, the newest history elements will be topmost on the stack, for quickest retrieval.

            Similarly, we will maintain a database for doctor information using a List. Each node of the list will store the doctor’s ID number, as well as a stack of all the patients seen by that doctor. The patient stack elements will hold the ID number of the patient, as well as the last date that patient was seen.  Each patient must appear only once in the stack, so if an old patient is seen again, then their entry is moved to the top of the stack, and the date is updated. Therefore, the most recently seen patients will be near the top of the stack.

            Both of these simple databases will be stored in files, named PatientDB.txt and DoctorDB.txt, respectively. The format for both of these will be similar. The first line of each file will hold the ID of the first patient/doctor in the DB. The next line will store the stack for that ID # on a single line. Additional entries will follow, using the same format. Below is an example of each.

Sample PatientDB.txt:

 

pt000001

      il847362 5/18/99 dr849388 il837434 4/23/83  dr837435

      pt834143   

      il243802 8/13/80 dr091834

 

Sample DoctorDB.tct

           

            dr849388

      pt000001 5/18/99 pt938434 4/22/99

      dr091834

pt837434 3/10/98 pt834143 8/13/80

                       

As you can see from the examples, the line for a stack stores the entire stack contents. Also, ID’s start with pt for patient, dr for doctor, or il for illness, followed by 6 digits.

 

            Once we have read our data from the appropriate files, we can begin processing.  Each patient in the queue must be seen by a doctor, and the appropriate records updated. So, to “process” a patient, we must first choose a doctor from the queue of available doctors.  However, we would prefer to have a patient seen by a doctor they are familiar with, so we will look at each doctor in the queue and compare them with those in the patient’s history. The first doctor we find in the queue that has seen the patient in the past, will be chosen to see the patient this time. If the patient is new, or has never been seen by any of our doctors, we will simply take the next available doctor.

            Once we have paired a doctor with a patient, then we will use a random number generator to determine the illness which that patient has. (Let’s hope this isn’t really the way they do it!) An illness will have the prefix il followed by 6 digits. (Let’s assume that there is a key somewhere to match the id’s of illnesses to the name of the illness, but we don’t need to implement this.) Once we have “diagnosed” the patient, then we need to update that patient’s history with the illness, current date, and doctor ID. If it’s a new patient, or they don’t have an entry in the DB, then one must be added for them.

Likewise, we must also update the records of the doctor who saw the patient. To do this, check the doctor’s patient stack to see if that doctor has seen this specific patient before.  If not, simply add the patient ID and current date to the top of the stack.  If they have seen the patient before, then the patient’s entry must be moved to the top of the stack and the date updated.

This process is repeated for the remaining patients, until the patient queue is empty. It’s interesting to note that if this system were to be used by a real ER, then rather than reading in a patient queue, the program would use real-time user I/O to input and process patients in the queue. For the sake of simplicity, however, we will postpone this feature until we have a hospital that wants to pay us to implement it.

The program should provide feedback to the user as it processes each patient.  First, the starting states of the DB’s and queues should be displayed, then, after each patient is processed, the complete info for that patient, the info for the doctor that saw them, and each queue should be displayed.

After all processing is complete, an updated PatientDB.txt, DoctorDB.txt, and DoctorQ.txt should all be written to disk, and the PatientQ.txt file should be emptied.
